  <action>
Create `.github/workflows/ci.yml` with the following structure. Use per-user decision: GitHub Actions, push to master + workflow_dispatch, four parallel jobs, pnpm caching, Node 20 only, no deployment.

Workflow-level configuration:
- name: CI
- triggers: push to branches [master] + workflow_dispatch
- env block: ANTHROPIC_API_KEY: test-key-ci and MAX_CONCURRENT_LLM: "3" (workflow-level so all jobs inherit)

Four parallel jobs (no depends_on between them), each with identical setup steps:
1. actions/checkout@v4
2. pnpm/action-setup@v4 with version: 10
3. actions/setup-node@v4 with node-version: 20 and cache: 'pnpm'
4. pnpm install --frozen-lockfile

Then each job runs its specific check:
- lint job (name: "Lint"): `pnpm run lint`
- typecheck job (name: "Typecheck"): `pnpm run typecheck`
- test job (name: "Test"): `pnpm run test`
- build job (name: "Build"): `pnpm run build`

All jobs use runs-on: ubuntu-latest.

CRITICAL: Use `master` branch in push trigger, NOT `main`. The project's default branch is master.

Per research: Do NOT use actions/setup-node@v6 (breaking caching changes). Use v4 which has stable `cache: 'pnpm'` support. Do NOT cache node_modules directly (fragile with pnpm symlinks). Do NOT use YAML anchors (not supported in GitHub Actions workflow files). The setup step duplication across four jobs is the standard pattern.
  </action>
