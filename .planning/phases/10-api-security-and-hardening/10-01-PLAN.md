  <behavior>

    - Rate limiter returns 429 after 10 requests/min to /schedules/parse
    - Rate limiter returns 429 after 60 requests/min to general endpoints like /agents
    - Rate limiter allows requests within the limit
    - Rate limiter tracks per-IP (different IPs have independent limits)
    - Cleanup timer removes stale IP entries
    - _resetRateLimiter clears state for test isolation
  </behavior>
  <action>
    1. Create `src/middleware/rate-limiter.ts`:
       - Module-level `requestLog = new Map<string, number[]>()` for sliding window timestamps per IP.
       - Helper `getClientIp(c: Context): string` -- check `c.req.header('x-forwarded-for')` first (split on comma, trim, take first), fallback to `(c.env as any)?.incoming?.socket?.remoteAddress ?? '127.0.0.1'`.
       - Helper `isRateLimited(ip: string, windowMs: number, maxRequests: number): boolean` -- filter timestamps within window, push current, set back in map, return `valid.length > maxRequests`.
       - Helper `isLlmEndpoint(path: string): boolean` -- returns true if path matches `/agents/` followed by any segment then `/execute`, OR path is `/schedules/parse`. Use simple string matching: `path.match(/^\/agents\/\d+\/execute$/)` or `path === '/schedules/parse'`.
       - Export `rateLimiterMiddleware(): MiddlewareHandler` -- get IP, determine limit (10/min for LLM endpoints, 60/min for others), check `isRateLimited`, return `c.json({ error: 'Rate limit exceeded' }, 429)` if exceeded, else `await next()`.
       - Cleanup: `setInterval` every 5 minutes, evict IPs whose newest timestamp is older than 2 minutes.
       - Export `stopRateLimiterCleanup()` to `clearInterval` for shutdown/tests.
       - Export `_resetRateLimiter()` that clears the Map (for test isolation).

    2. Update `src/index.ts` to mount middleware in correct order:
       - Add imports: `import { secureHeaders } from "hono/secure-headers"`, `import { cors } from "hono/cors"`, `import { authMiddleware } from "./middleware/auth.js"`, `import { rateLimiterMiddleware, stopRateLimiterCleanup } from "./middleware/rate-limiter.js"`.
       - Mount order after `app.use(logger())`:
         a. `app.use(secureHeaders({ xFrameOptions: 'DENY', referrerPolicy: 'same-origin' }))` -- security headers first so every response gets them
         b. `app.use(cors({ origin: () => '' }))` -- CORS before auth (preflight has no auth header)
         c. `app.use(rateLimiterMiddleware())` -- rate limit before auth (throttle brute force)
         d. `app.use(authMiddleware())` -- auth before routes
       - Note: Mount `secureHeaders` and `cors` directly in index.ts rather than importing from security.ts -- this keeps it simpler and matches the research pattern of using Hono built-ins directly. The `src/middleware/security.ts` file from Task 1 can still exist as a helper for testing, OR the security middleware can be configured inline in index.ts. Per research recommendation, inline is cleaner.
       - ALTERNATIVE (cleaner): Import `securityHeaders` and `corsMiddleware` from `./middleware/security.js` to keep index.ts clean. Use this approach.
       - Add `stopRateLimiterCleanup()` call in the `shutdown()` function before `server.close()`.

    3. Write `tests/middleware-rate-limiter.test.ts`:
       - Create a Hono test app with `rateLimiterMiddleware()` and test routes: `POST /agents/1/execute`, `POST /schedules/parse`, `GET /agents`.
       - Import and call `_resetRateLimiter()` in `beforeEach`.
       - Tests: allows requests within limit, blocks after 10 requests to execute endpoint, blocks after 10 requests to schedules/parse, blocks after 60 requests to general endpoint, different IPs tracked independently (use x-forwarded-for header to simulate).
       - Import and call `stopRateLimiterCleanup()` in `afterAll` to prevent timer leak.
