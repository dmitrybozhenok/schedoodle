  <name>Task 1: Enhanced health endpoint with per-agent breakdown, aggregates, upcoming runs, and status levels</name>
  <files>src/routes/health.ts, src/index.ts, tests/health.test.ts</files>
  <behavior>
    - Per-agent breakdown: each agent in `agents` array has agentId, agentName, lastRunAt, lastStatus, successRate (percentage), avgDurationMs, healthy, consecutiveFailures
    - successRate: percentage of successful executions in 24h window (0-100). Default 100 for agents with 0 executions in window. 'running' status excluded from calculation.
    - avgDurationMs: average duration of completed executions in 24h window. Default 0 for agents with 0 executions.
    - lastStatus: status of most recent execution (any time, not just 24h). null if no executions.
    - Upcoming runs: array of next 5 soonest scheduled runs across all agents, each with agentName and scheduledAt (ISO string), sorted by scheduledAt ascending
    - Top-level status: "ok" if all agents healthy AND circuit breaker closed; "degraded" if some agents unhealthy; "unhealthy" if circuit breaker OPEN OR more than half of agents are unhealthy
    - System-wide recentExecutions gains successRate (percentage) and avgDurationMs
    - Result/error in per-agent lastResult/lastError truncated to 200 chars with "..." suffix
    - All existing fields preserved: uptimeMs, agentCount, circuitBreaker, recentExecutions (success, failure, total)
    - createHealthRoute signature adds getScheduledJobs parameter
  </behavior>
  <action>
    1. **src/routes/health.ts**: Major rewrite of the health route. Update `createHealthRoute` signature to add a 4th parameter: `getScheduledJobs: () => Map<number, Cron>`. Import `Cron` type from croner, `agents` and `executionHistory` from schema, `desc`, `eq`, `gte`, `and`, `inArray` from drizzle-orm, and `getConsecutiveFailures` from enrich-agent helper.

    Add a `truncate(value: unknown, maxLength = 200): string | null` helper function inside the module:
    - If value is null/undefined, return null
    - Convert to string (JSON.stringify if not string)
    - If longer than maxLength, slice and append "..."

    Rewrite the GET "/" handler:

    **A. Compute per-agent stats:**
    - Fetch all agents from DB: `db.select().from(agents).all()`
    - Compute 24h cutoff: `new Date(Date.now() - 24*60*60*1000).toISOString()`
    - Fetch ALL executions in 24h window in a single query (bounded by time, not per-agent): `db.select().from(executionHistory).where(gte(executionHistory.startedAt, twentyFourHoursAgo)).all()`
    - Group executions by agentId in JavaScript (Map<number, Execution[]>)
    - For each agent, compute:
      - `lastRunAt`: query most recent execution (any time) for this agent -- use a single query fetching most recent execution per agent, or reuse the getLastRunAt helper
      - `lastStatus`: status of most recent execution (any time). Fetch alongside lastRunAt.
      - `lastResult`: result of most recent execution, truncated. `lastError`: error of most recent, truncated.
      - `successRate`: from 24h grouped executions, filter out 'running', compute `successCount / totalCompleted * 100`. Default 100 if no completed executions.
      - `avgDurationMs`: from 24h grouped executions where durationMs is not null, compute average. Default 0 if none.
      - `healthy` and `consecutiveFailures`: call `getConsecutiveFailures(agent.id, db)`, then `healthy = failures < 3`

    **B. Compute system-wide aggregates:**
    - From the same 24h executions (already fetched), compute overall success, failure, total counts (exclude 'running')
    - `successRate`: `successCount / totalCompleted * 100`, default 100 if no completed
    - `avgDurationMs`: average of all durationMs values (non-null), default 0

    **C. Compute upcoming runs:**
    - Call `getScheduledJobs()` to get the live Map<number, Cron>
    - Build an agent name lookup Map<number, string> from the agents list
    - Iterate jobs, call `job.nextRun()` for each, collect { agentName, scheduledAt: date.toISOString() }
    - Sort by scheduledAt ascending, take first 5

    **D. Compute top-level status:**
    - If circuit breaker state is "OPEN": return "unhealthy"
    - Count unhealthy agents from per-agent stats
    - If unhealthyCount === 0: "ok"
    - If unhealthyCount > agentStats.length / 2: "unhealthy"
    - Otherwise: "degraded"
    - Special case: if there are 0 agents, status is "ok" (nothing to be unhealthy)

    **E. Build response:**
    ```typescript
    return c.json({
      status,           // "ok" | "degraded" | "unhealthy"
      uptimeMs,
      agentCount,
      circuitBreaker,
      recentExecutions: {
        success,
        failure,
        total,
        successRate,     // NEW: number (percentage)
        avgDurationMs,   // NEW: number
      },
      agents: agentStats, // NEW: per-agent breakdown array
      upcomingRuns,       // NEW: next 5 upcoming runs
    });
    ```

    Note on N+1 concern: The per-agent `lastRunAt`/`lastStatus` and `consecutiveFailures` each require a small bounded query per agent. Given SQLite and expected agent counts (<100), this is acceptable. The 24h executions for successRate/avgDurationMs are fetched once and grouped in JS.
