    - isCronExpression("every weekday at 9am") returns false (NL text)

  <files>src/schemas/schedule-input.ts, src/services/schedule-parser.ts, tests/schedule-parser.test.ts</files>
  <behavior>
    - parseSchedule("every weekday at 9am") returns { cronExpression: "0 9 * * 1-5", humanReadable: "At 09:00, Monday through Friday", confidence: "high", interpretation: "..." }
    - parseSchedule("0 9 * * 1-5") detects cron input and returns description without LLM call (uses isCronExpression + cronstrue)
    - parseSchedule with ambiguous input returns confidence: "low"
    - parseSchedule throws when LLM fails after retry
    - cronstrue.toString is called on the generated cron expression for humanReadable field
    - Generated cron expression is validated with croner before returning
    - Invalid cron from LLM triggers retry with error feedback
  </behavior>
  <action>
    1. Create `src/schemas/schedule-input.ts` with:
       - `parseScheduleBody`: Zod schema for request body — `z.object({ input: z.string().min(1).max(500) })`
       - `scheduleParseSchema`: Zod schema for LLM structured output — `z.object({ cronExpression: z.string().describe("Standard 5-field cron expression (minute hour day-of-month month day-of-week)"), confidence: z.enum(["high", "low"]).describe("high if the input clearly maps to a single cron schedule, low if ambiguous"), interpretation: z.string().describe("Brief explanation of how the input was interpreted") })`
       - Export `ScheduleParseResult = z.infer<typeof scheduleParseSchema>`
       - Export `ParseScheduleResponse` type: `{ input: string; cronExpression: string; humanReadable: string; confidence: "high" | "low"; interpretation: string; warning?: string }`

    2. Create `src/services/schedule-parser.ts` with:
       - Import `generateText`, `NoObjectGeneratedError`, `Output` from "ai"
       - Import `Cron` from "croner", `cronstrue` from "cronstrue"
       - Import `DEFAULT_MODEL`, `resolveModel` from "../config/llm-provider.js"
       - Import `isCronExpression` from "../helpers/cron-detect.js"
       - Import `scheduleParseSchema` and types from "../schemas/schedule-input.js"
       - Import circuit breaker from executor: import `getLlmCircuitStatus` from "./executor.js" for status check, but for the actual LLM call, create a local function that calls `resolveModel` + `generateText` with `Output.object({ schema: scheduleParseSchema })`. Do NOT go through the executor's breaker — schedule parsing should use its own try/catch and NOT wrap in circuit breaker (the parse endpoint is a user-interactive query, not a background execution; it should get a clear error if LLM is down).
       - `buildParsePrompt(input: string): string` — builds the system prompt instructing the LLM to convert NL to cron. Include rules: use standard 5-field cron, set confidence to "low" when ambiguous, list common mappings (weekday=Mon-Fri, daily=every day, twice a day=0 9,17 default).
       - `parseSchedule(input: string): Promise<ParseScheduleResponse>` — the main export:
         a. If `isCronExpression(input)` is true, skip LLM, use `cronstrue.toString(input)` for humanReadable, return `{ input, cronExpression: input.trim(), humanReadable, confidence: "high", interpretation: "Input is already a valid cron expression" }`.
         b. Otherwise, resolve model with `DEFAULT_MODEL`, call `generateText` with `Output.object({ schema: scheduleParseSchema })` and the built prompt. Follow the retry pattern from executor.ts: catch `NoObjectGeneratedError`, retry once with error feedback appended.
         c. After getting the LLM result, validate the generated `cronExpression` with croner (`new Cron(expr, { paused: true }); job.stop()`). If croner throws, throw a new Error with message "LLM generated an invalid cron expression".
         d. Generate `humanReadable` using `cronstrue.toString(result.cronExpression)`.
         e. Return `ParseScheduleResponse` with all fields populated.
         f. Add `warning: "This interpretation may not match your intent. Please verify."` if confidence is "low".

       - Test: LLM returns low confidence — response includes warning field.

       - Test: NoObjectGeneratedError triggers retry with error feedback in prompt.
